What is Prim's Algorithm?

Prim's algorithm (1957) builds a Minimum Spanning Tree (MST) — a subset of edges that connects all vertices with minimum total weight and no cycles.

It grows from a single start vertex, always choosing the cheapest edge that crosses from the current tree into an unvisited vertex.

🔑 Design Paradigm: Greedy Prim’s is a Greedy Algorithm because at every step it makes the locally best choice: it always adds the cheapest safe edge that expands the current MST.
🔑 Greedy Choice (Cut Property) For any partition of vertices into two sets, the minimum-weight crossing edge is always safe to include in some MST. Prim's uses this idea at every step.
Step-by-Step Walkthrough

1. Set key[start] = 0, all others = ∞. Set parent = NIL for all.

2. Insert all vertices into a min-priority queue keyed by key[].

3. While PQ is not empty:

  a. Extract-Min: pull vertex u with smallest key — O(log V)

  b. Mark u as in-MST.

  c. For each neighbor v of u: if v ∉ MST and w(u,v) < key[v], do decrease-key: key[v] = w(u,v), parent[v] = u.

⚡ Complexity Breakdown V Extract-Min operations × O(log V) + up to E decrease-key operations × O(log V) = O(E log V).
🔎 Sorting / Searching Connection Prim’s is not a sorting algorithm, but it depends on a search structure: the min-priority queue (heap). This heap continuously “searches” for the smallest key using Extract-Min. Without this structure, the algorithm becomes slower (naïve version ≈ O(V²)).
Fully Worked Example

Graph: A–B(4), A–C(2), B–C(1), B–D(5), C–D(8), D–E(2)

Init: keys = {A:0, B:∞, C:∞, D:∞, E:∞}

Extract A → neighbors: set B→4, C→2

Extract C → B→min(4,1)=1 ✓ decrease-key, D→8

Extract B → D→min(8,5)=5 ✓ decrease-key

Extract D → E→2

Extract E

MST: A–C(2) + C–B(1) + B–D(5) + D–E(2) = 10

Common Mistakes to Avoid
❌ Picking globally cheapest edge That’s Kruskal’s behavior, not Prim’s. Prim’s only chooses edges crossing from the current tree to the outside.
❌ Skipping decrease-key updates If a cheaper connection is found, the key must be updated. Otherwise the MST can be wrong.
❌ Re-processing MST vertices Once extracted, a vertex is permanently in the MST and should not be updated again.
Worst-Case Preset Explanation

The Sparse / Worst preset is included to demonstrate a graph where Prim’s still must process many heap operations with little shortcut advantage.

Even though Prim’s remains correct, sparse graphs highlight that edge availability is limited, so the algorithm often progresses in a narrow step-by-step manner.

Why it matters This helps visualize how Prim’s behaves differently depending on graph density.

Compare with Kruskal on the Same Graph
• Prim’s: starts at one chosen vertex and expands outward.
• Kruskal’s: sorts all edges globally and picks the smallest safe ones.
• Both can produce the same total MST weight, but they may choose edges in a different order.

Why Greedy Works: The Cut Property
For any cut (partition of vertices into two sets S and V−S), the minimum-weight edge crossing the cut is always safe to add to an MST. Prim’s maintains exactly that cut between the current MST and the unvisited vertices.
What if we remove the heap-based searching step?
If we remove the min-priority queue, Prim’s must repeatedly scan all vertices to find the minimum key. The algorithm still works, but becomes slower — approximately O(V²) instead of O(E log V).
